# KBO Predictor 개선사항

> 작성일: 2026-05-30  
> 대상 문서: `doc/KBO_Prediction_Site_PRD.md`  
> 목적: 초기 기획과 현재 구현 상태의 차이를 정리하고, 공개 가능한 예측 서비스로 발전시키기 위한 우선순위를 정의한다.

---

## 1. 검토 요약

현재 프로젝트는 경기 일정 수집, 투수 시즌 성적 수집, ELO 기반 예측, 파크팩터, 날씨, 불펜 소진도, API, 프론트엔드 화면까지 초기 구현이 진행되어 있다.

다만 서비스 신뢰도를 좌우하는 아래 흐름은 아직 명확하게 완성되지 않았다.

```text
실제 데이터 수집
→ 예측 생성 및 공개
→ 경기 종료 후 결과 정산
→ 성과 집계
→ 모델 검증 및 개선
```

새 기능을 추가하기 전에 이 흐름을 안정화하는 것이 우선이다.

---

## 2. 핵심 개선 원칙

### 2-1. 예측은 수정 가능한 현재값이 아니라 공개 시점의 기록이어야 한다

현재 구조는 경기당 예측 결과를 1개만 저장하고, 재생성 시 기존 값을 갱신한다.

경기 전날 예측, 선발 발표 후 예측, 경기 직전 예측이 덮어써지면 과거에 공개한 승률을 검증할 수 없다. 사용자에게 공개한 예측은 불변 스냅샷으로 보존해야 한다.

### 2-2. 적중률보다 먼저 결과 정산의 신뢰성을 확보한다

적중률 화면이 있어도 경기 종료 후 실제 승자와 예측 결과를 자동으로 연결하지 못하면 의미가 없다.

경기 결과 수집 후 정산 작업이 자동 실행되고, 누락 경기와 실패 원인을 확인할 수 있어야 한다.

### 2-3. 예측 모델의 성능은 확률 품질까지 평가한다

승패 적중률만으로는 `51%` 예측과 `90%` 예측의 품질 차이를 알 수 없다.

Accuracy와 함께 Brier Score 또는 Log Loss, 확률 구간별 Calibration을 관리해야 한다.

### 2-4. 데이터가 없을 때의 동작을 제품 요구사항으로 정의한다

선발 투수, 날씨, 불펜 데이터가 없더라도 예측을 중단할지, 일부 지표만으로 폴백할지 명확히 정해야 한다.

화면에는 데이터 기준 시각과 누락 항목을 표시해야 한다.

## 3. P0: 공개 전 반드시 완료할 항목

### 3-1. 예측 스냅샷 저장

#### 문제

- 경기당 예측 1개만 저장된다.
- 재예측 시 기존 결과가 덮어써진다.
- 모델 버전은 기록되지만 예측 당시 입력 데이터와 공개 상태는 추적하기 어렵다.

#### 개선안

`prediction_runs` 테이블을 추가하고 예측 실행마다 새로운 행을 저장한다.

권장 필드:

| 필드 | 설명 |
|------|------|
| `id` | 예측 실행 ID |
| `game_id` | 대상 경기 |
| `generated_at` | 예측 생성 시각 |
| `published_at` | 사용자 공개 시각 |
| `prediction_type` | `preliminary`, `final`, `manual` |
| `model_version` | 모델 버전 |
| `home_win_prob` | 홈 승률 |
| `away_win_prob` | 원정 승률 |
| `predicted_winner_id` | 예측 승자 |
| `feature_snapshot` | 예측 당시 입력 지표 JSON |
| `key_factors` | 사용자에게 공개한 근거 JSON |
| `is_published` | 공개 여부 |

기존 `predictions` 테이블은 경기별 최신 공개 예측을 빠르게 조회하는 용도로 유지하거나, 최신 스냅샷 조회 뷰로 대체한다.

#### 완료 기준

- 같은 경기의 예측을 여러 번 생성해도 과거 기록이 유지된다.
- API가 최신 공개 예측과 과거 예측 목록을 구분해 반환한다.
- 경기 종료 후 공개되었던 예측값을 그대로 검증할 수 있다.

### 3-2. 경기 결과 정산 파이프라인

#### 문제

- 경기 결과 수집은 구현되어 있으나 `actual_winner_id`, `is_correct`를 갱신하는 정산 흐름이 명확하지 않다.
- 적중률 API가 실제 운영 데이터로 누적되지 않을 위험이 있다.

#### 개선안

경기 결과 수집 직후 아래 작업을 실행한다.

```text
final 경기 조회
→ 실제 승자 결정
→ 공개 예측 스냅샷 정산
→ 무승부·취소 경기 제외 처리
→ 집계 갱신
→ 미정산 경기 기록
```

정산 결과에는 `settled_at`, `settlement_status`, `settlement_reason`을 남긴다.

#### 완료 기준

- 종료된 경기가 자동 정산된다.
- 무승부, 취소, 연기 경기는 적중률 모수에서 제외된다.
- 미정산 경기 목록을 확인하고 재실행할 수 있다.
- 동일 경기를 재정산해도 중복 반영되지 않는다.

### 3-3. 경기 식별자 보강

#### 문제

현재 날짜와 홈·원정 팀 조합만으로 경기를 찾으면 더블헤더를 구분할 수 없다.

#### 개선안

- KBO 원천 데이터의 경기 ID를 `external_game_id`로 저장한다.
- 더블헤더 차수를 `doubleheader_no`로 저장한다.
- 원천 경기 ID가 없는 경우 날짜, 홈팀, 원정팀, 시작 시각, 더블헤더 차수를 조합한다.

#### 완료 기준

- 같은 날짜에 같은 팀 조합의 경기가 2개 있어도 별도 경기로 저장된다.
- 크롤러 재실행 시 중복 행이 생성되지 않는다.

### 3-4. 데이터 수집 실패 대응

#### 개선안

- 크롤러별 마지막 성공 시각 저장
- 실패 횟수와 오류 메시지 저장
- 네트워크 오류 재시도
- 파싱 결과가 0건이면 정상 완료로 처리하지 않음
- 수동 재수집 명령 제공
- 운영 화면 또는 로그에서 상태 확인

#### 완료 기준

- 데이터 누락과 “경기가 없는 날”을 구분할 수 있다.
- 수집 실패 후 재실행으로 복구할 수 있다.
- 예측 화면에서 데이터 기준 시각을 확인할 수 있다.

## 4. P1: 신뢰도 향상 항목

### 4-1. 성능 평가 지표 확대

관리할 지표:

| 지표 | 목적 |
|------|------|
| Accuracy | 단순 승패 적중률 |
| Brier Score | 확률 예측 오차 |
| Log Loss | 과도하게 확신한 오답 패널티 |
| Calibration | 예측 확률과 실제 승률 일치 여부 |
| Coverage | 전체 경기 중 예측 생성 비율 |
| Data Freshness | 입력 데이터 최신성 |

성과는 전체, 월별, 모델 버전별, 확률 구간별로 조회할 수 있어야 한다.

### 4-2. 데이터 누락 폴백 정책

예측 생성 시 사용 가능한 데이터 수준을 기록한다.

| 상태 | 사용 지표 | 화면 표시 |
|------|----------|----------|
| `full` | ELO, 선발, 흐름, 홈 이점, 파크팩터, 날씨, 불펜 | 최종 예측 |
| `partial` | 일부 지표 누락 | 참고용 예측, 누락 항목 표시 |
| `baseline` | ELO, 최근 흐름, 홈 이점 중심 | 초기 예측 |
| `unavailable` | 핵심 데이터 부족 | 예측 보류 |

### 4-3. 설명 가능성 보강

예측 당시 사용자에게 보여 준 근거를 스냅샷으로 저장한다.

예시:

```text
- ELO 레이팅 홈팀 +42 우위
- 홈 선발 ERA 2.81, 원정 선발 ERA 4.12
- 원정팀 불펜 최근 3일 12.1이닝 소화
- 우천 가능성으로 예측 신뢰도 하향
```

단순히 지표를 나열하지 말고, 어떤 방향으로 승률에 영향을 주었는지 표시한다.

### 4-4. 운영 상태 확인 기능

관리자용 최소 화면 또는 API를 추가한다.

- 오늘 경기 수
- 예측 생성 완료 수
- 결과 정산 완료 수
- 수집 실패 태스크
- 데이터별 마지막 성공 시각
- 미정산 경기
- 예측 누락 경기
- 현재 모델 버전

## 5. P2: PRD와 구현 상태 현행화

### 5-1. 데이터 소스 변경

초기 PRD의 Statiz 기반 설계를 수정한다.

| 데이터 | 기존 계획 | 현재 방향 |
|--------|----------|----------|
| 경기 일정·결과 | KBO 공식 | KBO 공식 AJAX API |
| 투수 시즌 성적 | Statiz | KBO 공식 기록 페이지 |
| 팀 스탯 | Statiz | 대체 소스 검토 또는 후순위 |
| 선발 라인업 | KBO 공식 | 미구현, 별도 분석 필요 |
| 날씨 | Open-Meteo | 유지 |

Statiz는 로그인 제약으로 자동 수집 소스로 사용하기 어렵다.

### 5-2. 모델 버전 현행화

초기 PRD는 ELO, 선발, 최근 흐름, 홈 이점을 중심으로 설명한다. 현재 구현에는 파크팩터, 날씨, 불펜 소진도가 포함되어 있다.

PRD에 `v2.0` 모델을 별도 기술하고, 지표별 가중치와 폴백 조건을 문서화한다.

### 5-3. 로드맵 상태 재작성

기존 로드맵은 모두 미완료 체크박스로 작성되어 있어 현재 상태를 판단하기 어렵다.

각 항목에 아래 상태를 사용한다.

| 상태 | 의미 |
|------|------|
| 완료 | 구현 및 기본 검증 완료 |
| 진행 중 | 구현되었으나 운영 검증 필요 |
| 보류 | 데이터 소스 또는 정책 결정 필요 |
| 미착수 | 아직 구현하지 않음 |

### 5-4. 기술 스택 정리

- Redis는 의존성과 컨테이너는 있으나 실제 캐싱 적용 범위를 명확히 정의한다.
- JWT는 관리자 기능이 생길 때 적용한다.
- 공개 조회 API는 인증 없이 제공한다.
- 프론트엔드 렌더링 전략은 실제 데이터 갱신 주기에 맞춰 재검토한다.

P0와 P1이 안정화된 뒤 진행한다.

- 알림 기능
- 모델 버전 A/B 비교
- 확률 구간별 신뢰도 시각화
- 사용자 관심 팀 설정

## 7. 테스트 전략

### 7-1. 필수 자동화 테스트

| 영역 | 테스트 항목 |
|------|------------|
| 크롤러 | 저장된 HTML·JSON fixture 파싱 |
| 업서트 | 동일 데이터 재수집 시 중복 방지 |
| 경기 식별 | 더블헤더 저장 |
| 예측 | 입력 누락 시 폴백 |
| 스냅샷 | 재예측 시 과거 기록 보존 |
| 정산 | 승패, 무승부, 취소, 재정산 |
| 백테스트 | 미래 데이터 참조 방지 |
| API | 최신 예측, 예측 이력, 적중률 응답 |

### 7-2. 백테스트 기준

- 시간순으로만 데이터를 적용한다.
- 경기 시작 이후에 확정된 데이터를 과거 예측에 사용하지 않는다.
- 모델 버전별 결과를 별도 보관한다.
- CSV 결과가 비어 있으면 성공으로 처리하지 않는다.
- Accuracy, Brier Score, Calibration을 함께 출력한다.

## 9. 전문 야구 사이트를 위한 데이터 및 UI 확장

### 9-1. 현재 수준 진단

현재 사이트는 “오늘 경기 예측 대시보드”로는 의미가 있지만, 프로야구 전문 사이트로 보기에는 데이터 범위가 좁다.

현재 확인되는 데이터:

| 영역 | 현재 제공 수준 | 한계 |
|------|---------------|------|
| 리그 | 팀 승패 기반 순위 | 게임차, 연속 기록, 홈·원정 성적 등 기본 순위표 항목 부족 |
| 팀 | 팀명, 구장, ELO, 최근 경기 | 시즌 공격·투수·수비 스탯과 선수단 없음 |
| 선수 | 일부 투수 기본 정보 | 전체 선수 명단, 타자 기록, 선수 상세, 경기별 로그 없음 |
| 경기 | 일정, 결과, 선발 일부, 예측 | 라인업, 박스스코어, 이닝별 득점, 경기 기록 없음 |
| 분석 | ELO, 최근 흐름, 선발, 파크팩터, 날씨, 불펜 | 원천 스탯과 계산 근거를 충분히 탐색하기 어려움 |

전문성을 갖추려면 사용자가 예측 결과를 본 뒤 팀, 선수, 경기 기록을 자연스럽게 탐색할 수 있어야 한다.

### 9-2. 데이터 확장 원칙

1. 공식 기록에 가까운 기본 데이터를 먼저 확보한다.
2. 원천 데이터와 계산 지표를 구분한다.
3. 시즌 누적 스탯과 경기별 로그를 함께 저장한다.
4. 데이터 출처와 마지막 갱신 시각을 화면에 표시한다.
5. 수집 가능 여부와 이용 조건을 확인한 소스만 운영에 사용한다.
6. 고급 지표는 계산식, 기준 시즌, 표본 수를 함께 표시한다.

## 10. Google AdSense 승인 준비 검토

### 10-1. 결론

현재 상태로 바로 AdSense 승인을 신청하는 것은 권장하지 않는다.

개인정보처리방침, 이용약관, 서비스 소개, 사이트맵, `robots.txt`, 광고 컴포넌트 등 기본 골격은 갖추고 있다. 다만 실제 콘텐츠가 예측 카드 중심이고 데이터가 없는 경우 빈 화면이 많아질 수 있다. Google이 요구하는 고유하고 유용한 게시자 콘텐츠와 충분한 사용자 경험을 안정적으로 제공한다고 보기에는 아직 이르다.

신청 전에 아래 P0 항목을 먼저 완료한다.

### 10-2. Google 공식 기준 요약

Google 공식 안내에서 중점적으로 확인하는 항목:

| 기준 | 확인 내용 |
|------|----------|
| 고유한 콘텐츠 | 방문자가 얻을 수 있는 독창적이고 유용한 정보가 있는가 |
| 충분한 게시자 콘텐츠 | 콘텐츠가 없거나 가치가 낮은 화면에 광고가 노출되지 않는가 |
| 명확한 탐색 | 메뉴와 링크가 실제 콘텐츠로 연결되고 사용하기 쉬운가 |
| 정책 준수 | 저작권 침해, 오인 유도, 기만적 표현이 없는가 |
| 개인정보 안내 | Google 광고 쿠키와 맞춤 광고 거부 방법을 안내하는가 |
| 광고 배치 | 콘텐츠보다 광고가 많거나 탐색 요소처럼 보이지 않는가 |

Google은 사이트 일부가 아니라 전체 사이트를 검토한다. 빈 페이지, 오류 페이지, 준비 중 페이지도 승인 판단에 영향을 줄 수 있다.

### 10-3. 현재 갖춘 항목

| 항목 | 상태 | 비고 |
|------|------|------|
| 개인정보처리방침 | 있음 | Google 광고 쿠키와 광고 설정 링크 포함 |
| 이용약관 | 있음 | 예측 정보의 참고 목적과 면책 안내 포함 |
| 서비스 소개 | 있음 | 모델 설명, FAQ, 데이터 출처 안내 포함 |
| 문의 수단 | 있음 | 이메일 공개 |
| `robots.txt` | 있음 | 전체 크롤링 허용 |
| 사이트맵 | 있음 | 정적 주요 페이지 등록 |
| 광고 컴포넌트 | 있음 | 홈 경기 카드 중간에 제한적으로 배치 |
| 광고 미설정 처리 | 있음 | 게시자 ID 또는 슬롯 미설정 시 광고 영역 미노출 |

기본 골격은 나쁘지 않다. 문제는 정책 페이지의 존재보다 실제 콘텐츠의 신뢰성과 밀도다.

### 10-4. 신청 전 P0 수정사항

#### 잘못되거나 검증되지 않은 설명 제거

현재 소개 페이지와 푸터에는 실제 구현과 맞지 않는 내용이 있다.

| 현재 노출 내용 | 문제 | 수정 방향 |
|---------------|------|----------|
| Statiz에서 팀·투수 성적 수집 | 현재 Statiz 자동 수집을 사용하지 않음 | KBO 공식 기록실로 변경 |
| 선발 라인업 확정 후 최종 예측 | 라인업 수집은 미구현 | 선발 정보 확보 수준에 따른 폴백 설명 |
| 백테스팅 적중률 약 `64%` | 현재 결과 파일과 검증 근거가 부족함 | 검증 완료 전 삭제 또는 “검증 중” 표시 |
| 데이터 소스 `3곳` | 실제 사용 범위와 다를 수 있음 | 운영 중인 소스만 표기 |
| 푸터의 Statiz 출처 | 현재 상태와 불일치 | 제거 또는 과거 사용 소스로 별도 설명 |

검증되지 않은 수치를 성과처럼 게시하면 사이트 신뢰도를 떨어뜨린다. 예측 서비스는 특히 모델 성능과 데이터 출처를 보수적으로 표현해야 한다.

#### 빈 페이지 최소화

현재 API 오류 또는 데이터 부족 시 여러 화면이 빈 상태가 될 수 있다.

- 홈: 경기 없는 날에도 순위 요약, 최근 결과, 분석 콘텐츠 제공
- 선발 투수: 선발 미등록 시 전체 투수 순위 또는 설명 콘텐츠 제공
- 예측 히스토리: 정산 데이터 확보 전에는 광고 없이 안내 콘텐츠 제공
- 팀 상세: 선수단과 팀 기록을 추가해 자체 콘텐츠 확보
- 경기 상세: 예정 경기와 종료 경기 모두 유용한 정보 제공

광고는 데이터가 없거나 콘텐츠가 짧은 화면에 배치하지 않는다.

#### 고유 콘텐츠 확대

단순히 KBO 기록을 재게시하는 수준으로 보이지 않도록 자체 분석을 추가한다.

- 경기별 예측 근거
- 팀별 ELO 변화와 최근 흐름 해설
- 확률 구간별 모델 성과
- 파크팩터와 날씨 영향 설명
- 야구 지표 용어 사전
- 데이터 수집 기준과 모델 변경 이력

원천 기록을 그대로 복제하지 말고, 자체 계산·비교·해설을 결합한다.

#### 링크와 데이터 출처 점검

- 존재하지 않는 Statiz 링크 제거
- KBO 공식 데이터 이용 조건 확인
- 팀 로고, 선수 사진, 기사 이미지 사용 시 권리 확인
- 데이터 출처와 마지막 갱신 시각 표시
- 서비스가 KBO 공식 서비스로 오인되지 않도록 비공식 분석 서비스임을 명시

### 10-5. 광고 배치 개선사항

#### 권장 원칙

- 광고는 게시자 콘텐츠와 명확하게 구분한다.
- 메뉴, 버튼, 경기 카드처럼 보이게 만들지 않는다.
- 빈 상태, 오류 페이지, 준비 중 페이지에는 광고를 노출하지 않는다.
- 광고보다 콘텐츠가 충분히 많아야 한다.
- 초기 승인 단계에서는 광고 슬롯을 적게 유지한다.

#### 현재 배치 평가

홈 경기 카드 4번째 이후의 배너 1개는 과도하지 않다. 다만 경기가 적은 날에는 광고가 표시되지 않아도 괜찮다. 승인 목적 때문에 빈 공간이나 정책 페이지에 광고를 억지로 늘리지 않는다.

#### 추가 권장 사항

- 광고 컨테이너 상단에 작은 `광고` 레이블 표시
- 콘텐츠 로딩 실패 시 광고 요청 방지
- 정책, 약관, 오류, 빈 상태 페이지에는 광고 미배치
- 모바일에서 광고가 콘텐츠를 밀어내거나 가리지 않는지 확인

### 10-6. 개인정보 및 동의 관리

현재 개인정보처리방침에는 Google AdSense 쿠키와 광고 설정 링크가 포함되어 있다. 아래를 추가 점검한다.

- 실제로 Google Analytics를 사용하지 않는다면 관련 문구 제거
- Analytics를 사용할 경우 실제 스크립트와 개인정보 안내를 일치시킴
- EEA, 영국, 스위스 사용자에게 광고를 제공할 경우 Google 요구사항에 맞는 인증 CMP 적용
- AdSense 관리 화면의 Privacy & messaging 설정 검토
- 향후 추가 광고 네트워크 사용 시 개인정보처리방침 갱신

### 10-7. `ads.txt`와 도메인

`ads.txt`는 현재 저장소에 없다.

- AdSense 게시자 ID 발급 후 `frontend/public/ads.txt` 추가
- AdSense에서 안내하는 판매자 행을 정확히 기재
- 배포 후 `/ads.txt` 접근 확인
- 가능하면 Vercel 기본 주소보다 자체 도메인 사용
- 사이트맵, `robots.txt`, 메타데이터의 기본 URL을 실제 도메인으로 통일

자체 도메인은 승인 필수 조건으로 단정할 수는 없지만, 운영 신뢰성과 URL 일관성 측면에서 권장한다.

### 10-8. 스포츠 예측 콘텐츠 주의사항

현재 서비스는 스포츠 분석과 통계 예측을 목적으로 하고, 약관과 푸터에 도박 목적 활용 금지를 표시하고 있다. 이 방향을 유지한다.

Google Publisher Restrictions에는 온라인 도박 관련 콘텐츠가 포함된다. 따라서 아래 기능은 추가하지 않는다.

- 배팅 사이트 링크
- 배당률 비교
- 가입 추천 코드
- 배팅 업체 제휴 배너
- 수익 보장 표현
- “확실한 픽”, “무조건 적중” 같은 과장 문구

사이트의 중심은 배팅 추천이 아니라 야구 데이터, 통계 해설, 모델 검증이어야 한다.

2. 주요 페이지에서 빈 화면과 오류가 없다.
3. 잘못된 데이터 출처와 검증되지 않은 성능 표현을 제거했다.
4. 개인정보처리방침, 이용약관, 문의 수단, 도메인 설정이 실제 운영 상태와 일치한다.
5. 광고가 콘텐츠보다 많지 않고, 광고와 탐색 요소가 명확히 구분된다.

승인 여부는 Google 검토 결과에 따라 달라지므로 보장할 수 없다. 다만 위 조건을 충족하면 현재보다 승인 가능성과 장기 운영 안정성이 높아진다.

## 11. 결론

현재 우선순위는 기능 확장이 아니라 검증 가능한 운영 루프 완성이다.

공개한 예측을 변경 없이 보존하고, 경기 종료 후 자동 정산하며, 확률 품질을 측정할 수 있어야 서비스의 핵심 가치인 “데이터 기반 근거와 신뢰도 공개”가 성립한다.

동시에 프로야구 전문 사이트로 발전하려면 예측 결과만 보여 주는 구조에서 벗어나야 한다. 전체 선수단, 타자·투수 기록, 팀 기록, 일정·결과, 경기 상세 데이터를 확보하고, 사용자가 리그에서 팀과 선수까지 자연스럽게 탐색할 수 있는 UI를 제공해야 한다.
