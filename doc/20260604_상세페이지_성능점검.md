# KBO Predictor 상세 페이지 성능/UX 점검

기준일: 2026-06-04 ~ 2026-06-05

## 배경

Render 배포 화면에서 경기 카드를 클릭한 뒤 상세 화면 진입이 약 10초까지 느려지는 문제가 있었다. 로컬에서도 `/games/{id}` 진입 시 404 또는 hydration/runtime error처럼 보이는 증상이 함께 확인됐다.

초기 원인은 하나가 아니라 다음 요소가 겹친 상태였다.

- 상세 화면 첫 진입 시 무거운 API를 한 번에 기다림
- 종료 경기에서 예측 로딩 카드가 영구 표시되는 분기 오류
- 로컬 Docker/Next dev 서버의 `.next` 캐시 충돌
- 확정 타순 원천 데이터가 `null`일 때 빈 탭처럼 보이는 UX 문제
- 일부 문서/문구 인코딩이 깨져 유지보수성이 낮아진 문제

## 완료된 개선

### 1. 상세 화면 초기 로딩 개선

상세 첫 화면은 가벼운 요약 데이터를 먼저 보여주고, 무거운 상세 데이터는 이후에 로드하도록 조정했다.

- 첫 화면에서 경기 헤더와 기본 예측을 빠르게 노출
- 선발/공격력/예측 분석은 필요한 시점에 자연스럽게 표시
- 프론트 로컬 개발 환경에서 파일 변경 감지를 안정화하기 위해 Docker polling 옵션 적용

### 2. 종료 경기 예측 로딩 버그 수정

종료 경기에서 `prediction && !isFinished` 조건 때문에 예측 카드가 계속 로딩 상태로 남는 문제가 있었다.

수정 후 종료 경기에서는 다음을 보여준다.

- 경기 전 예측 승률
- 실제 결과 기준 적중/빗나감
- 무승부일 경우 별도 안내

### 3. `타순` 탭을 `공격력` 탭으로 개편

확정 타순은 네이버 스포츠 원천 API가 공개해야만 수집된다. 실제로 2026-06-05 경기의 네이버 라인업 API는 `lineUpData: null`을 반환했고, 로컬 DB에서도 해당 날짜 경기의 라인업 행은 0건이었다.

그래서 타순을 핵심 탭으로 두는 대신, 항상 보여줄 수 있는 `공격력` 탭으로 개편했다.

현재 공격력 탭에서 보여주는 정보:

- 최근 7경기 성적
- 평균 득점
- 평균 OPS
- 평균 안타
- 평균 홈런
- 볼넷/삼진 비율
- 경기당 볼넷/삼진
- 상대 선발 공략 포인트
- 확정 타순 발표 대기/반영 상태

### 4. 예측 분석 탭 개선

사용자가 “왜 이 팀이 유리한가?”를 빠르게 이해하도록 예측 분석 탭 상단을 보강했다.

추가된 정보:

- 예측 요약 문장
- 데이터 완성도
- 승률 바
- 요인별 반영/대기 체크리스트
- 아직 대기 중인 데이터
- 승률 변화 타임라인

### 5. 데이터 상태 카드 추가

프리뷰 탭에 데이터 상태 카드를 추가했다.

표시 내용:

- 반영 완료 데이터
- 대기/주의 데이터
- 개별 데이터 갱신 상태
- 마지막 예측 갱신 시각
- 확정 타순 미발표 시 안내 문구

확정 타순이 없을 때는 사용자가 빈 화면으로 오해하지 않도록, 공격력 탭의 최근 타격 흐름과 상대 선발 공략 지표를 참고하라는 안내를 제공한다.

## 현재 판단

상세 화면은 이전보다 사용자에게 더 빨리 의미 있는 정보를 제공한다. 특히 확정 타순이 없을 때도 `공격력`, `선발`, `예측 분석`, `데이터 상태`를 통해 분석 흐름을 유지할 수 있게 됐다.

다만 Render 환경의 상세 API 자체가 느려질 가능성은 남아 있다. 화면 로딩 UX는 개선됐지만, 백엔드 계산이 무거운 구간은 별도로 측정할 필요가 있다.

## 남은 리스크

### 1. 확정 타순 원천 데이터 의존성

타순은 네이버 원천 API가 공개하지 않으면 수집할 수 없다. 따라서 모델 핵심 지표보다는 보조 지표로 유지하는 것이 안전하다.

권장 방향:

- 확정 타순은 있으면 반영
- 없으면 공격력 탭의 최근 팀 타격 흐름으로 대체
- UI에서는 항상 발표 전/대기 상태를 명확히 표시

### 2. Render API 응답 속도

uptime health check는 `/v1/health`만 깨우므로 상세 API의 무거운 계산을 직접 빠르게 만들지는 못한다.

추가 확인이 필요한 API:

- `/v1/games/{id}`
- `/v1/games/{id}/prediction`
- `/v1/games/{id}/summary`

### 3. 문구/인코딩 품질

화면 주요 문구는 정상화됐지만, 과거 커밋에서 만들어진 일부 주석이나 문서에는 깨진 문자열이 남아 있을 수 있다. 사용자 화면에 노출되는 문구를 우선으로 정리하고, 내부 주석은 기능 변경 시 함께 정리하는 편이 안전하다.

## 다음 작업 후보

### 1. 성능 측정만 먼저 진행

소스 구조를 바꾸기 전에 API별 응답 시간을 기록한다.

- 로컬 `/v1/games/{id}` 응답 시간
- Render `/v1/games/{id}` 응답 시간
- `/prediction` 응답 시간
- DB 쿼리 또는 계산 병목 후보

### 2. 중복 섹션 정리

프리뷰와 예측 분석 탭에 데이터 상태/승률 변화 정보가 일부 겹칠 수 있다. 사용자가 반복으로 느끼는지 확인한 뒤, 필요하면 한쪽은 요약형으로 줄인다.

### 3. 불펜 UI 접기/펼치기

불펜 소모 명단은 유용하지만 길어질 수 있다. 기본은 요약만 보여주고, 사용자가 펼쳤을 때 투수별 로그를 보게 하면 화면 밀도가 좋아진다.

### 4. Render 내부 API URL 점검

프론트 서버가 백엔드를 공인 URL로 호출하고 있다면 Render 내부 네트워크 URL 또는 서비스 URL 사용을 검토한다.

목표:

- 외부 왕복 제거
- 상세 SSR/API 호출 지연 완화
- 로컬/웹 화면 차이 감소
