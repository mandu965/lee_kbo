# KBO Predictor 추가 개선사항

> 작성일: 2026년 5월 31일  
> 기준: 모델 `v2.1`, 전체 실테스트 완료 이후 코드·문서 재검토  
> 관련 문서: `doc/20260530_개선사항.md`, `doc/20260531_전체실테스트.md`

## 1. 결론

현재 시스템은 단순 승률 카드 수준을 벗어났다.

- KBO 일정·결과, 투수·타자 시즌 기록, 순위 수집
- 종료 경기 박스스코어와 최근 7경기 OPS
- 선발 최근 5경기
- 불펜 최근 3일 선수별 등판 상세와 부상 상태
- 확정 타순, 선수별 OPS, 라인업 강도, 제외·대체 선수
- 예측 스냅샷, 정산, Brier Score, 변화 타임라인
- 로컬·원격 DB 동기화와 전체 통합 배치

다음 단계의 우선순위는 기능 수를 늘리는 것이 아니다.

```text
운영 배치 안정화
→ 예측 변수 검증
→ 선수·팀 탐색 깊이 강화
→ 경기 상세 콘텐츠 확장
→ SEO·AdSense 운영 준비
```

## 2. 배포 전 우선 개선

### P0-1. 스케줄러를 API 프로세스에서 분리

#### 현재 상태

FastAPI lifespan에서 APScheduler를 시작한다.

```text
API 시작
→ scheduler.start()
```

로컬 단일 워커에서는 동작하지만 배포 인스턴스나 웹 워커가 2개 이상이면 같은 배치가 동시에 실행될 수 있다. 2026년 5월 31일 실테스트에서 ELO 중복 반영을 발견해 DB 제약으로 차단했지만, 모든 배치가 같은 수준으로 보호되는 것은 아니다.

#### 개선안

- 수집·예측 스케줄러를 별도 worker 서비스로 분리
- 배치 시작 시 PostgreSQL advisory lock 또는 Redis 분산 락 획득
- 동일 작업의 중복 실행 방지 키 사용: `task_name + target_date`
- API 서버는 조회와 관리자 실행 요청만 담당

#### 완료 기준

- API 인스턴스를 2개 실행해도 배치 작업은 1번만 수행
- worker 재시작 후 누락 작업을 재실행 가능
- 중복 실행 시 로그에 `skipped: lock held` 기록

### P0-2. 수집 작업 실행 이력과 실패 알림

#### 현재 상태

크롤러는 로그를 남기지만 마지막 성공 시각, 수집 건수, 실패 원인을 DB에서 조회할 수 없다. 외부 HTML 구조 변경으로 파싱 결과가 `0건`이 되어도 운영자가 즉시 인지하기 어렵다.

#### 개선안

`collection_runs` 테이블과 관리자 상태 API를 추가한다.

| 필드 | 설명 |
|------|------|
| `task_name` | 일정, 결과, 투수, 타자, 순위, 타순, 날씨, 박스스코어 |
| `target_date` | 대상 날짜 |
| `started_at`, `finished_at` | 실행 구간 |
| `status` | `running`, `success`, `warning`, `failed` |
| `row_count` | 저장 건수 |
| `error_message` | 실패 사유 |
| `retry_count` | 재시도 횟수 |

추가로 네트워크 오류에는 제한된 횟수의 exponential backoff를 적용한다. 평일 경기일인데 결과·일정·순위가 `0건`이면 성공이 아니라 경고로 처리한다.

#### 완료 기준

- `/v1/admin/collection-status`에서 마지막 성공 시각과 건수 확인
- 실패 시 이메일, Discord 또는 Slack 알림
- 수동 재실행 결과도 같은 이력에 저장

### P0-3. 데이터 최신성 표시

#### 현재 상태

화면은 예측 생성 시각을 표시하지만, 투수·타자·순위·타순·날씨가 각각 언제 갱신됐는지 사용자가 판단하기 어렵다.

#### 개선안

- 경기 상세 상단에 `데이터 기준 시각` 표시
- 지표마다 `updated_at`, 원천, 표본 수 표시
- 일정 시간 이상 오래된 입력에는 `갱신 지연` 배지 표시
- 타순 미발표, 날씨 미수집, 불펜 로그 부족을 구분

### P0-4. 예측 스냅샷 중복 억제

#### 현재 상태

배치를 수동 재실행하면 입력값이 같아도 `prediction_runs`가 계속 추가된다. 스냅샷 보존은 필요하지만 동일 입력의 반복 저장은 변화 타임라인을 불필요하게 복잡하게 만든다.

#### 개선안

- `feature_snapshot` 정규화 후 `input_hash` 저장
- `game_id + model_version + prediction_type + input_hash`가 같으면 신규 스냅샷 생략
- 수동 실행은 `manual` 타입으로 구분
- `baseline`, `starter_confirmed`, `weather_updated`, `lineup_confirmed`, `final` 생성 조건 명확화

## 3. 모델 신뢰성 개선

### P1-1. 예비 파크팩터의 승률 반영 중단

#### 현재 상태

코드 주석에는 파크팩터가 `UI 검증용 예비 추정치`라고 명시되어 있지만 예측 엔진에서는 `5%` 가중치를 가진다.

#### 개선안

- 검증 전에는 화면 참고 지표로만 표시
- 모델 기여도는 `0%p`, `available=false` 처리
- 최근 2~3시즌 구장별 득점·홈런 데이터를 수집
- 홈·원정 공격력, 시즌 환경 변화, 표본 수를 고려해 재산출
- 백테스트에서 Brier Score와 Calibration 개선이 확인된 뒤 재활성화

### P1-2. 정상 타선 기준 고도화

#### 현재 상태

정상 타선은 시즌 타석 수 상위 9명이다. 공격력 대용 지표로는 유용하지만 포지션을 무시하므로 포수, 유격수, 외야수 구성에 따라 제외·대체 판정이 과도하게 발생할 수 있다.

#### 개선안

- 최근 14일 선발 출전 횟수와 포지션을 함께 반영
- 포지션 그룹별 정상 주전 산출
- 부상자와 엔트리 제외 상태 연결
- 제외 선수 OPS 손실량과 대체 선수 OPS 차이를 직접 계산
- 포지션 데이터가 부족하면 현재 방식으로 폴백

### P1-3. 모델 검증 지표 확대

#### 현재 상태

Accuracy와 Brier Score 일부를 저장하지만 모델 버전별 비교, Calibration, Coverage, Log Loss를 운영 화면에서 확인하기 어렵다.

#### 개선안

| 지표 | 목적 |
|------|------|
| Accuracy | 승패 적중률 |
| Brier Score | 확률 오차 |
| Log Loss | 과도한 확신 패널티 |
| Calibration | `60%` 예측이 실제로 약 `60%` 적중하는지 확인 |
| Coverage | 전체 경기 중 정상 예측 비율 |
| Freshness | 최신 입력을 사용한 예측 비율 |

집계 축은 `전체`, `월별`, `모델 버전별`, `prediction_type별`, `확률 구간별`로 나눈다.

### P1-4. 백테스트 재현성 확보

- 실운영 스냅샷과 백테스트 입력 구조 통일
- 모델 버전, 데이터 기준 시각, 결측 폴백을 함께 저장
- 미래 데이터가 과거 예측에 섞이지 않는지 검사
- 모델 가중치 변경 전후 Brier Score 비교 보고서 생성

## 4. 프로야구 전문 사이트 데이터 확장

### P2-1. 타자 경기별 로그

#### 필요성

현재 타자는 시즌 누적 기록과 종료 경기 팀 OPS 중심이다. 선수 단위 최근 폼을 설명하려면 타자 경기별 로그가 필요하다.

#### 추가 데이터

- 경기별 타석, 타수, 안타, 2루타, 3루타, 홈런
- 볼넷, 삼진, 타점, 득점
- 최근 7경기 OPS
- 최근 14일 OPS
- 장타율·출루율 변화

### P2-2. 이닝별 스코어와 실책

#### 현재 상태

`team_game_stats.errors` 필드는 있으나 실제 박스스코어 수집에서 채우지 않는다. 이닝별 득점도 저장하지 않는다.

#### 개선안

- 경기별 `line_scores` 저장
- 팀별 실책, 잔루, 득점권 성과 수집 검토
- 종료 경기 상세에 박스스코어 탭 추가
- 최근 경기의 초반·후반 득점 비중 분석

### P2-3. 팀 상세 페이지 강화

#### 현재 상태

팀 상세는 승패, ELO 차트, 최근 경기 목록 중심이다.

#### 추가 UI

- 팀 로스터: 투수, 포수, 내야수, 외야수
- 팀 공격: AVG, OBP, SLG, OPS, 홈런, 볼넷, 삼진
- 팀 투수: ERA, WHIP, 선발 ERA, 불펜 ERA
- 홈·원정 스플릿, 월별 성적, 최근 10경기
- 부상·엔트리 변동
- 최근 경기와 예정 경기 링크

### P2-4. 선수 상세 페이지

#### 현재 상태

선수 순위표는 있지만 선수명에서 상세 페이지로 이동할 수 없다. 투수 API 일부는 존재하지만 타자 상세 화면은 없다.

#### 추가 UI

- `/players/{id}` 선수 상세
- 프로필, 소속팀, 포지션, 부상 상태
- 시즌 누적 기록
- 최근 경기 로그 차트
- 월별 성적과 홈·원정 스플릿
- 타자는 최근 OPS, 투수는 최근 ERA·WHIP

### P2-5. 상대 전적과 스플릿

우선 참고 지표로 제공하고 모델 가중치는 바로 부여하지 않는다.

- 팀 간 시즌 상대 전적
- 홈·원정 성적
- 월별 성적
- 좌·우완 상대 OPS
- 선발 투수의 상대 팀 기록
- 표본 수와 함께 표시

## 5. 경기 상세 UI 개편

현재 경기 상세는 정보량이 증가하면서 세로 길이가 길어졌다. 다음 구조로 탭을 분리한다.

```text
프리뷰 | 타순 | 박스스코어 | 선수 기록 | 예측 분석
```

| 탭 | 핵심 내용 |
|----|----------|
| 프리뷰 | 승률, 핵심 변화, 데이터 완성도, 선발, 불펜 요약 |
| 타순 | 선수별 OPS, 정상 타선 대비 제외·대체, 부상 상태 |
| 박스스코어 | 종료 경기 이닝별 득점, 안타, 실책 |
| 선수 기록 | 선발 최근 5경기, 주요 불펜 3일 로그, 타자 최근 폼 |
| 예측 분석 | 요인별 `%p`, 변화 타임라인, 결측 데이터, 모델 버전 |

홈 경기 카드에는 가장 큰 승률 변화 요인 1개만 추가한다.

예:

```text
라인업 반영 후 한화 +1.8%p
불펜 소진으로 LG -1.2%p
```

## 6. AdSense·SEO 운영 준비

### P3-1. 운영 도메인 확정

현재 `robots.txt`는 `https://kbo-predictor.vercel.app/sitemap.xml`을 고정 참조한다. 운영 도메인이 정해지면 같은 호스트 기준으로 현행화해야 한다.

Google Search 공식 문서에 따르면 `robots.txt`의 sitemap 항목은 프로토콜과 호스트를 포함한 완전한 URL이어야 한다.

### P3-2. 동적 sitemap

현재 sitemap은 정적 페이지 위주다.

추가 대상:

- `/teams/{id}`
- `/games/{id}`
- `/players/{id}`
- 최근 경기 URL

검색 노출 대상으로 삼을 URL만 포함한다. 빈 페이지, 데이터가 부족한 페이지는 제외한다.

### P3-3. `ads.txt`와 개인정보 메시지

- AdSense 게시자 ID 발급 후 루트 도메인의 `/ads.txt` 제공
- 루트 도메인에서 파일 접근 가능 여부 확인
- AdSense `Privacy & messaging` 설정 검토
- 개인정보처리방침의 실제 광고 설정과 문구 일치 확인

Google AdSense 공식 문서에 따르면 `ads.txt`는 루트 도메인에서 접근 가능해야 하며 반영에는 시간이 걸릴 수 있다.

### P3-4. 고유 콘텐츠 확대

승률 카드만으로는 콘텐츠 깊이가 약하다.

- 경기별 데이터 기반 프리뷰 문단
- 팀별 시즌 분석 페이지
- 선수 상세 페이지
- 용어 사전 내부 링크
- 예측 결과 회고와 모델 변경 기록

자동 생성 문장은 실제 지표와 표본을 근거로 만들고 과장 표현을 피한다.

## 7. 기술 부채 정리

### P4-1. 문서와 실제 코드 정합성

PRD에는 분석 엔진 `v2.0`, Redis 예측 캐시, JWT 인증이 기재되어 있으나 현재 코드와 맞지 않는 부분이 있다.

- 모델 버전을 `v2.1`로 현행화
- Redis와 JWT를 실제 도입할지 제거할지 결정
- 완료된 체크리스트 상태 갱신
- 데이터 원천을 KBO 공식, 네이버 스포츠, Open-Meteo 기준으로 정리

### P4-2. 투수 시즌 기록과 경기 로그 테이블 분리

현재 `pitcher_stats`가 시즌 누적 행과 경기 로그 행을 `game_id IS NULL` 여부로 구분한다.

권장 구조:

- `pitcher_season_stats`
- `pitcher_game_logs`

쿼리 조건 누락으로 시즌 기록과 경기 로그가 섞일 위험을 줄인다.

### P4-3. 외부 경기 ID 제약 강화

- `games.external_game_id`의 유일성 검토
- 원천 ID가 존재하는 행은 partial unique index 적용
- 더블헤더 테스트 케이스 추가

### P4-4. 테스트 범위 확대

현재 자동 테스트는 라인업 보정 중심이다.

추가 테스트:

- ELO 멱등성과 동시 실행
- 결과 정산 재실행
- 일정 크롤러 더블헤더
- 박스스코어 파싱
- 타순 미발표 폴백
- 불펜 3일 경계값
- API 계약 테스트
- 프론트 핵심 화면 smoke test

CI에서는 아래 명령을 필수로 실행한다.

```bash
pytest -q
alembic check
npm run lint
npm run build
```
