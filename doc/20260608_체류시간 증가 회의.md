# 2026-06-08 KBO 시스템 체류시간 증가 회의

## 회의 목적

KBO 예측 시스템의 체류시간을 늘리기 위해 먼저 실행할 기능을 정리한다.

이번 회의에서는 **체류시간/이탈 측정 고도화는 우선순위를 뒤로 미루고**, 사용자가 바로 더 오래 머물 가능성이 높은 다음 2개 과제를 먼저 진행한다.

1. 홈 화면 경기 중심 대시보드 개선
2. 경기 상세 페이지 탐색 깊이 강화

## 우선순위 재정리

## 1순위. 홈 화면 경기 중심 대시보드 개선

### 왜 먼저 해야 하는가

홈 화면은 대부분 사용자의 첫 진입 지점이다. 현재 홈은 오늘 경기 목록과 메뉴 진입 중심이므로, 사용자가 어떤 경기를 눌러야 하는지 판단하는 데 필요한 정보가 부족하다.

홈에서 바로 “오늘 어떤 경기가 볼 만한지”, “왜 이 경기를 봐야 하는지”, “어디를 누르면 더 깊게 볼 수 있는지”가 보여야 상세 페이지 진입률과 체류시간이 늘어난다.

### 목표

- 홈 화면에서 경기 상세 페이지 진입률 증가
- 사용자가 오늘 경기의 핵심 정보를 한 화면에서 파악
- 주목 경기, 접전 경기, 선발 매치업이 좋은 경기로 자연스럽게 이동
- 세션당 페이지뷰 증가

### 구현 대상 파일

- `frontend/app/page.tsx`
- `frontend/components/GameCard.tsx`
- `frontend/lib/api.ts`
- `frontend/lib/types.ts`
- 필요 시 `backend/app/routers/games.py`
- 필요 시 `backend/app/schemas/game.py`

### 홈 화면 구성안

홈 화면은 다음 순서로 재구성한다.

1. 오늘의 경기 요약
2. 오늘의 주목 경기
3. 경기 카드 목록
4. 순위/선수/블로그 보조 링크

### 1. 오늘의 경기 요약

홈 상단에 오늘 경기 상태를 요약한다.

표시 항목:

- 전체 경기 수
- 예정 경기 수
- 진행 중 경기 수
- 종료 경기 수
- 가장 접전인 경기
- 예측 확률 차이가 가장 큰 경기

구현 방법:

- `getTodayGames()` 결과를 기반으로 프론트에서 계산한다.
- 경기 상태는 `scheduled`, `in_progress`, `finished` 기준으로 분류한다.
- 예측 확률이 있는 경우 `abs(home_win_prob - away_win_prob)`로 접전도를 계산한다.
- 예측 확률이 없는 경기는 선발 정보 또는 최근 흐름이 있는 경기만 보조 후보로 둔다.

UI 방향:

- 큰 설명문보다 숫자와 경기명을 중심으로 배치한다.
- 모바일에서도 첫 화면 안에 경기 요약과 첫 경기 카드가 보여야 한다.
- “오늘의 접전 경기”를 클릭하면 해당 경기 상세로 이동한다.

### 2. 오늘의 주목 경기

경기 목록 위에 “오늘의 주목 경기”를 별도 섹션으로 둔다.

선정 기준:

- 예측 확률 차이가 작은 경기
- 선발 투수 정보가 양쪽 모두 있는 경기
- 예측 데이터 완성도가 높은 경기
- 최근 흐름 차이가 뚜렷한 경기
- 진행 중 경기인 경우 우선 노출

표시 항목:

- 홈팀 vs 원정팀
- 경기 시간/구장
- 예측 승률
- 주목 이유 1줄
- 핵심 변수 2개
- 상세 분석 버튼

주목 이유 예시:

- `예측 확률 차이 2.4%p 접전`
- `양 팀 선발 정보 반영 완료`
- `최근 5경기 득실 흐름 차이 큼`
- `라인업 반영 후 홈 승률 상승`

구현 방법:

- 초기에는 프론트에서 계산한다.
- 추후 백엔드 응답에 `highlight_reason`, `key_factors`, `data_completeness`를 추가한다.

프론트 계산 예시:

```ts
const spread = Math.abs(game.prediction.home_win_prob - game.prediction.away_win_prob);
const isCloseGame = spread <= 0.05;
const hasStarters = !!game.starters?.home && !!game.starters?.away;
```

### 3. 경기 카드 강화

현재 `GameCard`를 더 탐색형 카드로 강화한다.

추가할 정보:

- 승률 막대
- 선발 투수 비교
- 핵심 변수 2개
- 최근 5경기 흐름
- 데이터 완성도
- 경기 상태 배지

CTA:

- 분석 보기
- 선발 비교
- 예측 변화

구현 방법:

- `GameCard`에 `variant` prop을 추가한다.
  - `default`
  - `featured`
  - `compact`
- 홈의 주목 경기는 `featured`를 사용한다.
- 상세 페이지 하단의 다른 경기 추천에는 `compact`를 사용한다.

예상 구조:

```tsx
<GameCard game={game} variant="featured" />
<GameCard game={game} variant="compact" />
```

### 4. 홈 보조 링크 재배치

팀 순위, 선수 기록, 일정, 블로그 링크는 유지하되 홈 상단보다 아래로 내린다.

이유:

- 사용자의 첫 행동을 “경기 상세 진입”으로 유도해야 한다.
- 메뉴형 링크는 탐색 보조 역할로 충분하다.

### 백엔드 확장안

초기에는 프론트 계산으로 시작한다. 이후 API 품질을 높이기 위해 games 응답에 다음 필드를 추가한다.

- `prediction_summary`
- `highlight_reason`
- `key_factors`
- `data_completeness`
- `starter_matchup_summary`
- `recent_trend_summary`

백엔드에서 계산하면 좋은 이유:

- 홈, 상세, 블로그에서 같은 요약 문구를 재사용할 수 있다.
- 프론트의 조건문을 줄일 수 있다.
- 향후 SEO 메타데이터에도 활용할 수 있다.

### 우선 구현 순서

1. 홈에서 오늘 경기 요약 계산
2. 오늘의 주목 경기 섹션 추가
3. `GameCard`에 featured/compact variant 추가
4. 경기 카드에 승률 막대와 핵심 변수 표시
5. 메뉴 링크를 하단 보조 영역으로 재배치
6. 필요 시 games API 응답 확장

### 성공 기준

- 홈에서 경기 상세로 이동하는 클릭 증가
- 홈 이탈률 감소
- 홈에서 사용자가 최소 1개 경기 카드를 클릭하도록 유도
- 모바일 첫 화면에서 오늘의 핵심 경기 확인 가능

## 2순위. 경기 상세 페이지 탐색 깊이 강화

### 왜 두 번째로 해야 하는가

홈에서 사용자를 상세 페이지로 보내도, 상세 페이지 안에서 볼거리가 부족하면 체류시간은 크게 늘지 않는다.

현재 경기 상세에는 예측, 라인업, 투수, 분석 탭이 있으므로 완전히 새로 만들 필요는 없다. 기존 데이터 구조를 활용해 “비교”, “변화”, “다른 경기 이동”을 강화하는 것이 핵심이다.

### 목표

- 경기 상세 평균 체류시간 증가
- 탭 전환 수 증가
- 경기 상세에서 다른 경기 상세로 이동 증가
- 예측 근거를 더 쉽게 이해하도록 개선

### 구현 대상 파일

- `frontend/app/games/[id]/page.tsx`
- `frontend/components/GameDetailClient.tsx`
- `frontend/components/GameDetailTabs.tsx`
- `frontend/components/GameCard.tsx`
- `frontend/lib/api.ts`
- `frontend/lib/types.ts`
- 필요 시 `backend/app/routers/games.py`
- 필요 시 `backend/app/routers/predictions.py`

### 상세 페이지 강화 방향

상세 페이지는 다음 4개 영역을 강화한다.

1. 승부처 요약
2. 예측 변화 타임라인
3. 선수/투수 매치업 비교
4. 오늘의 다른 경기 추천

### 1. 승부처 요약

경기 상세 상단 또는 분석 탭 최상단에 “승부처 요약”을 배치한다.

표시 항목:

- 예측에 영향을 준 핵심 변수 상위 3개
- 각 변수가 어느 팀에 유리한지
- 몇 %p 영향을 줬는지
- 사용자가 이해할 수 있는 짧은 설명

예시:

- 선발 우위: LG +4.2%p
- 불펜 피로: 두산 -2.1%p
- 최근 득실 흐름: LG +1.8%p

구현 방법:

- 기존 `prediction.factor_contributions`를 활용한다.
- `available === true`인 항목만 사용한다.
- `Math.abs(contribution_pp)` 기준으로 정렬한다.
- 상위 3개를 카드 형태로 표시한다.

설명 문구 생성 예:

```ts
function factorNote(label: string, contribution: number, teamName: string) {
  return `${label} 지표에서 ${teamName} 쪽 우위가 반영되었습니다.`;
}
```

### 2. 예측 변화 타임라인

현재 상세 페이지에는 예측 변화 데이터인 `prediction.trend`를 활용할 여지가 있다. 이 부분을 더 눈에 띄게 만든다.

표시 단계:

- 초기 예측
- 선발 반영
- 라인업 반영
- 날씨 반영
- 최종 예측

각 단계 표시 항목:

- 생성 시각
- 예측 타입
- 홈 승률
- 원정 승률
- 직전 단계 대비 변화폭
- 데이터 완성도

구현 방법:

- `prediction.trend`를 시간순으로 정렬한다.
- `change_pp`가 있는 항목은 상승/하락 색상을 다르게 표시한다.
- 변화폭이 큰 항목은 강조한다.
- 모바일에서는 세로 타임라인으로 보여준다.

강조 기준:

- 3%p 이상 변화: 강조
- 5%p 이상 변화: 강한 강조

### 3. 선수/투수 매치업 비교

사용자는 비교형 콘텐츠에 오래 머문다. 실제 상대전적 데이터가 부족하더라도 현재 보유한 선발, 팀 최근 흐름, 라인업 데이터를 조합해 1차 구현이 가능하다.

비교 항목:

- 선발 투수 ERA, WHIP, K/9, BB/9, HR/9
- 최근 평균 이닝
- 최근 5경기 팀 득실차
- 라인업 OPS
- 불펜 피로도

구현 방법:

- 이미 `GameDetailClient.tsx`에 선발/불펜/최근 흐름 관련 UI가 있으므로 별도 섹션으로 재배치한다.
- 중복 설명을 줄이고, 비교 테이블 형태로 정리한다.
- 우위 값은 색상으로 표시한다.

UI 구성:

- 좌측: 원정팀
- 중앙: 지표명
- 우측: 홈팀

예:

| 원정 | 지표 | 홈 |
|---:|---|:---|
| 3.42 | 선발 ERA | 3.88 |
| 1.18 | WHIP | 1.31 |
| +7 | 최근 득실 | -2 |

### 4. 오늘의 다른 경기 추천

상세 페이지 하단에 “오늘의 다른 경기”를 추가한다.

목표:

- 상세 페이지에서 바로 이탈하지 않도록 다음 경기 링크 제공
- 한 세션에서 여러 경기 상세를 보게 만들기

추천 기준:

- 현재 경기 제외
- 진행 중 경기 우선
- 예정 경기 우선
- 예측 접전 경기 우선
- 데이터 완성도 높은 경기 우선

구현 방법:

- 상세 페이지에서 오늘 경기 목록을 추가로 가져온다.
- 현재 경기 ID를 제외한다.
- 3개만 노출한다.
- `GameCard variant="compact"`를 사용한다.

주의:

- 상세 페이지 하단 추천은 너무 큰 카드로 만들지 않는다.
- 사용자가 현재 경기 내용을 다 본 뒤 자연스럽게 이동할 수 있게 배치한다.

### 5. 탭 구조 정리

현재 탭이 많아지면 사용자가 어디를 봐야 할지 모를 수 있다.

추천 탭:

- 프리뷰
- 선발/라인업
- 승부처
- 예측 변화

기존 분석 탭 안에 너무 많은 정보가 들어가 있다면 다음처럼 재배치한다.

- 승부처: factor contribution 중심
- 예측 변화: trend 중심
- 선발/라인업: 선수 비교 중심
- 프리뷰: 전체 요약 중심

### 우선 구현 순서

1. `factor_contributions` 기반 승부처 요약 추가
2. `prediction.trend` 기반 예측 변화 타임라인 강화
3. 선발/불펜/최근 흐름 비교 UI 정리
4. 상세 페이지 하단 오늘의 다른 경기 추천 추가
5. 탭 구조 정리
6. 필요 시 백엔드 응답 필드 보강

### 성공 기준

- 상세 페이지 내 탭 전환 증가
- 상세 페이지 하단 추천 경기 클릭 증가
- 상세 페이지에서 홈으로 되돌아가는 대신 다른 경기로 이동
- 예측 근거를 이해하기 쉬워졌다는 정성 피드백 확보

## 3순위. 체류시간/이탈 측정 고도화

### 우선순위를 미루는 이유

측정 고도화는 중요하지만, 현재 단계에서는 먼저 사용자가 실제로 더 볼 만한 화면을 만드는 것이 더 시급하다.

따라서 1차로 홈과 상세 페이지를 개선한 뒤, 다음 단계에서 측정 체계를 고도화해 개선 효과를 검증한다.

### 나중에 구현할 항목

- 페이지별 평균 체류시간
- 경기 상세 탭별 체류시간
- 스크롤 깊이

## 4순위 이후 과제

### 내 팀/관심 선수 개인화

초기에는 로그인 없이 `localStorage`로 구현한다.

저장 항목:

- `favorite_team_id`
- `favorite_player_ids`

홈 화면에 내 팀 오늘 경기, 내 팀 최근 흐름, 내 팀 다음 경기를 보여준다.

### 경기 전/후 자동 콘텐츠 강화

경기 전에는 프리뷰, 경기 후에는 리뷰를 자동 생성한다.

콘텐츠 타입:

- `preview`
- `lineup_update`
- `final_prediction`
- `review`

경기 상세와 블로그 페이지에서 연결한다.

### 예측 결과 회고 및 적중률 콘텐츠

히스토리 페이지를 강화한다.

표시 항목:

- 최근 7일 적중률
- 최근 30일 적중률
- 확률 구간별 적중률
- 경기별 예측 성공/실패 회고

### 참여형 기능

초기 후보:

- 팬 승부 예측 투표
- 경기 MVP 투표
- 데일리 퀴즈

### 알림/재방문 장치

초기에는 웹 푸시보다 사이트 내부 배너부터 구현한다.

예:

- 내 팀 경기가 30분 뒤 시작합니다.
- 라인업이 반영되어 예측이 변경되었습니다.
- 경기 리뷰가 생성되었습니다.

## 회의에서 결정할 사항

1. 홈 화면에서 가장 강하게 보여줄 기준
   - 접전 경기
   - 진행 중 경기
   - 데이터 완성도 높은 경기
   - 인기 팀 경기

2. 경기 카드에 반드시 넣을 정보
   - 승률 막대
   - 선발 투수
   - 핵심 변수
   - 최근 흐름
   - 데이터 완성도

3. 경기 상세 탭 구조
   - 기존 탭 유지 후 섹션만 강화
   - 탭 이름과 구성을 재정리

4. 백엔드 확장 범위
   - 1차는 프론트 계산으로 진행
   - 1차부터 games API 응답을 확장

## 추천 결론

이번 회의의 추천 결론은 다음과 같다.

1. 체류시간/이탈 측정 고도화는 뒤로 미룬다.
