# KBO Predictor 상세 페이지 성능 점검 메모

기준일: 2026-06-04

## 배경

- Render 배포 화면에서 경기 카드 클릭 후 상세 페이지 진입 체감이 약 10초 수준으로 느렸음
- 로컬에서는 `/games/{id}` 진입 시 404처럼 보이는 증상이 있었음

## 실제 원인

### 1. 첫 진입에 너무 많은 데이터를 기다리고 있었음

기존 상세 페이지는 화면 진입 직후 아래 두 API를 모두 기다리고 있었음.

- `/v1/games/{id}`
- `/v1/games/{id}/prediction`

이 두 API에는 다음과 같은 무거운 정보가 포함되어 있었음.

- 최근 폼
- 선발 최근 기록
- 라인업
- 데이터 신선도
- 불펜 피로도
- 날씨
- 파크팩터
- 예측 추세

즉, 사용자는 “상세 상단 헤더”를 보기 위해서도 무거운 분석 데이터를 함께 기다리는 구조였음.

### 2. 로컬 404의 실제 원인은 백엔드 500이었음

로컬에서는 `teams.home_elo`, `teams.away_elo` 컬럼이 없어 상세 API가 500을 내고 있었음.

- 겉으로는 404처럼 보였음
- 실제로는 DB 스키마 불일치 문제였음

### 3. uptime health check와 상세 API 성능은 별개였음

`/v1/health`를 uptime으로 주기적으로 호출해도, 무거운 상세 API는 여전히 느릴 수 있음.

즉:

- 서버가 “자고 있지 않음”
- 상세 API가 “빠름”

이 두 가지는 같은 뜻이 아님.

## 적용한 개선

### 1. 상세 첫 화면 경량화

새 요약 엔드포인트를 추가함.

- `/v1/games/{id}/summary`

이 엔드포인트는 아래 정보만 빠르게 반환하도록 구성함.

- 상단 경기 카드
- 팀명 / 날짜 / 상태
- 최근 폼
- 기본 예측
- 간단 선발 정보

### 2. 탭별 지연 로딩

상세 페이지를 다음처럼 분리함.

- `프리뷰`: 요약 데이터만으로 즉시 렌더링
- `타순`: 탭 진입 시 로딩
- `선발`: 탭 진입 시 로딩
- `예측 분석`: 탭 진입 시 로딩

즉, 전체 상세를 한 번에 완성하지 않고 필요한 시점에만 무거운 데이터를 불러오도록 변경함.

### 3. 백그라운드 프리로드

첫 화면이 뜬 뒤 약 0.6초 후, `선발/타순`용 상세 데이터를 백그라운드에서 선요청하도록 보완함.

의도:

- 사용자가 프리뷰를 먼저 보는 동안
- 다음 탭에 필요한 데이터를 미리 받아
- 탭 클릭 시 대기 시간을 줄이기 위함

### 4. 링크 prefetch 재조정

초기에는 상세 페이지 전체가 무거웠기 때문에 prefetch가 오히려 부담이 되었음.

그래서 한때 비활성화했지만, 상세 첫 화면을 가볍게 나눈 뒤에는 다시 활성화함.

현재 의도:

- 목록에서 경기 카드가 보일 때
- 가벼운 상세 셸을 미리 받아두고
- 실제 클릭 시 체감 속도를 개선

### 5. 에러 처리 개선

- 진짜 404와 백엔드 500을 구분하도록 수정
- 로컬에서는 DB 컬럼 추가와 컨테이너 재시작으로 500 원인 제거

## 검증 결과

### 로컬

- `/v1/games/291/summary`: 약 0.036초
- `/games/291` 첫 응답: 약 0.33 ~ 0.39초

### Render

첫 바이트 기준:

- 상세 페이지 첫 바이트: 약 0.49초
- 재요청 시 첫 바이트: 약 0.28 ~ 0.37초

여전히 느린 API:

- `/v1/games/{id}`: 약 9 ~ 11초
- `/v1/games/{id}/prediction`: 약 5 ~ 8초
- `/v1/games/{id}/summary`: 첫 요청 기준 4초대가 나오는 경우 있음

## 현재 판단

- 가장 큰 문제였던 “상세 첫 진입 대기”는 상당 부분 개선됨
- 사용자는 이전보다 훨씬 빨리 상세 화면의 기본 뼈대를 보게 됨
- 다만 Render 백엔드의 무거운 상세 API 자체는 여전히 느림

즉 현재 구조는:

- 첫 화면은 빠르게
- 무거운 분석은 뒤로 미루는 방향

으로는 적절함.

## 남은 보완 포인트

### 1. Render 내부 네트워크 URL 적용

가장 큰 다음 개선 후보.

- 프론트 서버가 백엔드를 공인 URL 대신 내부 URL로 호출하도록 `INTERNAL_API_URL` 설정 권장
- 코드상 우선순위는 이미 `INTERNAL_API_URL ?? NEXT_PUBLIC_API_URL`

즉 Render 환경변수만 넣으면 바로 효과를 볼 수 있는 구조임.

### 2. 무거운 상세 API 추가 경량화

- `/v1/games/{id}`에서 최근 기록 / 라인업 / 데이터 신선도 조립을 더 나누기
- `/v1/games/{id}/prediction`에서 불펜 계산 결과 캐시 검토
- 필요한 경우 더 작은 전용 엔드포인트로 세분화

### 3. 탭별 엔드포인트 추가 분리 가능

현재는 구현 복잡도와 체감 성능 사이에서 절충한 상태.

필요하면 다음처럼 더 쪼갤 수 있음.

- `/games/{id}/lineup`
- `/games/{id}/pitchers`
- `/games/{id}/analysis`
